[VirtualBox/DB] 인프라 보안 수업 - MariaDB 계정 보안 및 복제(Replication) 구축

날짜: 2026년 4월 9일
주제: MariaDB, 권한 관리(Grant/Revoke), DB 이중화(Replication)

📝 오늘 배운 내용 요약

1. MariaDB 계정 보안 및 권한 체계

  • 유저 생성 (Create User): 단순히 아이디만 만드는 것이 아니라 사용자@접속위치를 하나의 세트로 관리하는 개념을 익혔습니다. (localhost는 서버 내부 전용, %는 외부 전체 허용)
  • 권한 부여 (Grant): ALL PRIVILEGES는 모든 권한을 주지만, 실무 환경에서는 보안을 위해 SELECT, INSERT 등 필요한 권한만 최소한으로 골라 부여하는 것이 핵심임을 배웠습니다.
  • 권한 회수 및 삭제: 부여한 권한을 다시 뺏는 법(REVOKE)과 계정을 완전히 삭제하는 절차(DROP USER)를 실습했습니다.

2. DB 복제(Replication)의 메커니즘

  • 바이너리 로그(Binary Log): Master 서버에서 일어나는 모든 데이터 변화를 기록하는 ‘블랙박스’ 역할을 학습했습니다.
  • 릴레이 로그(Relay Log): Slave 서버가 Master의 기록을 가져와 임시로 저장하는 ’복사본 일기장’의 개념을 이해했습니다.
  • 동기화 과정: Master의 로그 번호(File)와 줄 번호(Position)를 Slave에게 알려주어, 해당 지점부터 실시간 데이터 추적 및 동기화가 시작되는 원리를 파악했습니다.

💭 오늘의 회고

배운 점

  • 정교한 문법: identified를 identifide로 친 오타 하나가 전체 쿼리 실행을 막는 것을 보며, SQL 문의 엄격함을 배웠습니다.
  • 인코딩의 중요성: 데이터베이스 생성 시 utf8mb4 설정을 누락하면, 한글 데이터 입력 시 Error 1366이 발생하며 시스템이 멈춘다는 것을 직접 확인하고 인코딩의 중요성을 체감했습니다.

트러블슈팅 (어려운 점 및 개선할 점)

  • Conflicting Read Lock (에러 1223): Master 상태를 확인하기 위해 걸어둔 잠금(Read Lock)을 UNLOCK TABLES로 풀지 않으면, Master 본인도 데이터를 쓸 수 없다는 점이 까다로웠습니다.
  • 로그 위치 불일치 (에러 1236): Master의 IP가 바뀌거나 재시작될 때 로그 번호가 변하는데, 이를 Slave에 즉시 업데이트해주지 않으면 복제가 중단되는 현상을 겪었고 이를 디버깅하는 과정이 어려웠습니다.

💡 액션 플랜 및 함께 나누고 싶은 점

  • 체크리스트 활용: 복제가 안 될 때는 당황하지 않고 SHOW SLAVE STATUS\G;를 실행해 Slave_IO_Running과 Slave_SQL_Running 상태가 모두 YES인지 확인하는 루틴을 체화하겠습니다.
  • 에러 건너뛰기: 데이터 정합성보다 서비스 유지가 우선인 특수한 상황에서 SQL_SLAVE_SKIP_COUNTER를 사용해 막힌 에러를 우회하는 실무적인 방법을 숙달하겠습니다.
  • 성취감 공유: 숱한 에러를 뚫고 복제 설정 상태 창에 YES/YES가 동시에 뜨는 순간 느꼈던 엄청난 쾌감과 성취감을 잊지 않고 앞으로의 학습 원동력으로 삼겠습니다.
