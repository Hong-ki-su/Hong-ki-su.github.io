[GNS3/보안] 인프라 보안 수업 - Cisco ASA 방화벽 기반 네트워크 보안 실무

날짜: 2026년 5월 6일
주제: GNS3, Cisco ASA 방화벽, OSPF 동적 라우팅, Stateful Inspection

📝 오늘 배운 내용 요약

1. GNS3 VM 기반 네트워크 가상화 환경 구축

  • 인프라 구성: VMware Workstation 17 Player를 활용하여 GNS3 VM(버전 2.2.57)을 설치 및 구동했습니다.
  • 리소스 최적화: GNS3 클라이언트와 VM을 연동하고 적절한 리소스(vCPU: 2, RAM: 2048MB)를 할당하여 안정적인 방화벽 실습 환경을 구성했습니다.

2. Cisco ASA 방화벽 기본 인터페이스 및 라우팅 설정

  • 방화벽 초기 설정: 방화벽 인터페이스에 IP를 할당하고, nameif (inside/outside) 설정 및 security-level (0, 100) 개념을 적용하여 신뢰 구간과 비신뢰 구간을 논리적으로 분리했습니다.
  • 동적 라우팅: 라우터(R1, R2)와 방화벽 간의 OSPF (Area 0) 동적 라우팅 프로토콜을 구성하고 인접성(Neighbor)을 맺었습니다.

3. 방화벽 보안 정책(ACL) 이해 및 트러블슈팅

  • Stateful Inspection: Security-Level이 높은 곳에서 낮은 곳으로의 일반 트래픽은 기본 허용하되, 방화벽의 특성상 차단되는 ICMP에 대해 예외 처리를 진행했습니다.
  • Ping 상태 추적: fixup protocol icmp 명령어를 통해 ICMP 패킷에 대한 상태 추적(Stateful) 기능을 활성화했습니다.
  • OSPF 단절 해결: 방화벽 외부(Outside) 인터페이스에 ACL을 적용할 때 발생하는 OSPF 단절 문제를 인지하고, permit ospf any any 구문을 통해 라우팅 프로토콜을 예외 처리했습니다.

💭 오늘의 회고

배운 점

  • 라우터는 경로만 있으면 패킷을 통과시키지만, 방화벽은 명시적인 허용 정책(ACL)이나 상태 추적 기능이 없으면 자기 자신을 향한 Ping조차 철저하게 차단한다는 점(Deny All 기본 원칙)을 실무 수준에서 뼈저리게 체감했습니다.
  • show arp 명령어를 통해 물리적 연결 상태와 IP 설정의 정확성을 교차 검증하는 L2/L3 트러블슈팅 기법을 익혔습니다.

트러블슈팅 및 아찔했던 순간

  • IP 충돌 실수: 라우터 인터페이스에 IP를 할당할 때 다른 포트와 서브넷이 겹쳐 overlap 에러가 발생하며 IP가 제대로 입력되지 않는 실수를 겪었습니다.
  • 라우팅 테이블 증발: 방화벽에 확장 접근 제어 목록(Extended ACL)을 적용하자, Implicit Deny (암묵적 차단) 규칙에 의해 OSPF Hello 패킷까지 모두 차단되어 라우팅 테이블(O 경로)이 통째로 날아가는 아찔한 상황을 겪었습니다.
  • 개선점: 설정 명령어를 무작정 입력하기 전에, show ip interface brief나 show ip route를 통해 현재 인프라 상태를 먼저 점검하는 조심스러운 습관이 부족했음을 반성했습니다.

💡 액션 플랜 및 실무 다짐

  • 논리적 병목 탐색: 통신 장애 발생 시 명령어를 지우고 다시 쓰기보다는, 통신 구간을 쪼개어 단계별 Ping 테스트(PC → 게이트웨이 → 방화벽 내부 → 방화벽 외부)를 진행하며 병목 구간을 정확히 찾아내는 논리적 접근법을 습관화하겠습니다.
  • 사전 예외 처리 체크리스트: 방화벽에 ACL을 적용하기 전, OSPF 동적 라우팅이나 관리용 SSH 등 인프라 유지에 필수적인 프로토콜을 간과하지 않도록 나만의 ’사전 예외 처리 체크리스트’를 구축하겠습니다.

🤝 함께 나누고 싶은 점

  • 설정이 완벽하다고 생각했는데도 통신이 안 될 때는, 방화벽의 보안 정책(ACL) 문제인지, 아니면 라우팅 경로(show ip route) 자체가 끊어진 것인지를 명확히 분리해서 생각하는 것이 문제 해결의 최우선 핵심이라는 점을 동기들과 공유하고 싶습니다.
