날짜: 2026년 5월 7일
주제: pfSense, OpenVPN, Cisco ASAv, Firewall ACL, OSI 7계층 트러블슈팅

📝 오늘 배운 내용 상세 요약

1. pfSense 기반 OpenVPN 서버 구축 및 클라이언트 환경 구성

  • 사용자 및 패키지 관리 심화:
    • System → User Manager 메뉴를 통해 VPN 접속을 위한 전용 계정을 생성하고, 인증서(Certificate) 기반의 안전한 암호화 통신을 위해 전용 인증서를 직접 발급하여 계정과 연동했습니다.
    • System → Package Manager 메뉴에 접근하여 클라이언트 배포를 위한 필수 확장 패키지(OpenVPN Client Export 등)를 성공적으로 설치했습니다.
  • VPN 서버 세팅 및 클라이언트 배포:
    • VPN → OpenVPN 설정에서 터널 네트워크 대역 및 암호화 알고리즘 등 세부 파라미터를 설정하여 VPN 서버 데몬을 활성화했습니다.
    • 설정 완료 후 외부 접속용 .ovpn 클라이언트 설정 파일을 추출하여 실제 클라이언트 환경에 배포하는 과정을 거쳤습니다.
  • 인프라 상태 점검 및 실전 접속 테스트:
    • OpenVPN GUI 프로그램을 통해 외부(Windows) 환경에서 내부 VPN Server로의 실제 터널링 접속 테스트를 수행했습니다.
    • Status → Services 대시보드에서 시스템 핵심 데몬(openvpn, dhcpd, dpinger 등)의 정상 작동 여부를 꼼꼼히 확인했습니다.
    • System → Routing → Gateways 메뉴를 통해 WAN 인터페이스 및 VPN 터널의 게이트웨이 라우팅 상태까지 완벽하게 점검 완료했습니다.

2. Cisco ASAv 방화벽 네트워크 인프라 구축 및 접근 제어(ACL)

  • 네트워크 아키텍처 설계: 내부망(Inside), 외부망(Outside), DMZ 망으로 논리적/물리적 구간이 구분된 가상 인프라 아키텍처를 설계했습니다.
  • 보안 레벨 및 상태 기반 통신 제어: 방화벽의 상태 기반 검사(Stateful Inspection) 원리와 각 인터페이스별 보안 레벨(Security Level: 0~100)을 적용하여, 높은 보안 영역에서 낮은 보안 영역으로 흐르는 트래픽을 정밀하게 제어했습니다.
  • 접근 제어 목록(ACL) 실무 구현: 특정 단말기의 트래픽 종류(Ping, Telnet, HTTP, SSH)를 식별하고, 이를 세밀하게 허용(permit)하거나 차단(deny)하는 방화벽 ACL 정책을 실무 인프라 수준으로 직접 구현했습니다.

💭 오늘의 회고 및 트러블슈팅

📌 배운 점

  • 오픈소스 방화벽인 pfSense를 활용해 실제 기업 환경과 유사한 VPN 서버를 직접 구축해 보았습니다. 이를 통해 단순한 계정 생성을 넘어 ‘인증서 기반의 암호화 통신’ 원리와 패키지 관리자의 유용성을 깊이 있게 이해할 수 있었습니다.
  • 방화벽 통신의 대원칙인 ’높은 보안 레벨에서 낮은 레벨로의 트래픽 흐름’과, 방화벽이 나갔던 패킷의 상태를 기억하여 응답을 허용하는 상태 기반 검사(Stateful Inspection) 원리를 확실하게 정립했습니다.
  • 설정 완료 후 단편적인 Ping 테스트에 그치지 않고, 대시보드(Services, Gateways)를 모니터링하여 전체적인 서비스와 라우팅 상태를 시각적으로 교차 검증하는 시스템 엔지니어의 점검 방법을 터득했습니다.

🚨 아찔했던 트러블슈팅 (어려운 점 및 해결 과정)

  • [L1 계층 및 암호화 알고리즘 이슈 해결]
    • 논리적인 명령어 설정(IP, 라우팅, 방화벽 정책)을 완벽하게 마쳤음에도 GNS3(EVE-NG) 인프라 통신 단절 이슈가 발생했습니다. 원인을 집요하게 분석한 결과, L1 계층인 물리적 케이블이 잘못된 포트(Gi0/0 대신 Management 등)에 연결되어 발생한 기초적인 문제임을 확인하고 도면에 맞게 재결선하여 단번에 해결했습니다.
    • 최신 리눅스에서 구형 방화벽으로 SSH 접속 시 보안 알고리즘 세대 차이(diffie-hellman, ssh-rsa)로 인해 연결이 거부되는 암호화 오류 현상을 겪었으나, 당황하지 않고 암호화 우회 옵션 명령어를 적용하여 성공적으로 원격 접속을 이끌어냈습니다.
  • [pfSense Web GUI 초기 접속 이슈 및 방화벽 우회]
    • pfSense의 강력한 기본 보안 정책(WAN 트래픽 전체 차단)으로 인해 웹 관리자 페이지 접속에 큰 애를 먹었습니다.
    • CLI 쉘 환경에서 pfctl -d 명령어를 통해 방화벽을 강제 비활성화했음에도 접속이 안 되는 복합 장애가 발생했습니다. 하지만 포트(HTTP/HTTPS) 문제 및 브라우저 보안 캐시 문제를 하나씩 추적(Ping을 통한 네트워크 단절 유무 선행 파악)하였고, 결국 쉘에서 접속 포트를 영구 허용하는 easyrule pass 명령어를 활용하여 관리자 접속 문제를 완벽하게 해결해 냈습니다.

💡 실무를 위한 액션 플랜

  • OSI 7계층 기반의 논리적 트러블슈팅 습관화: 네트워크 통신에 문제가 생겼을 때 무작정 명령어를 다시 입력하기보다, L1(물리적 케이블) → L3(IP 및 라우팅/게이트웨이) → L4/L7(포트, 보안 알고리즘, 방화벽 ACL) 순으로 체계적인 디버깅을 진행하는 엔지니어의 습관을 들이겠습니다.
  • 나만의 트러블슈팅 지식 자산화: 오늘 겪은 방화벽 접속 에러 해결법(SSH 레거시 알고리즘 우회 명령어, pfSense 쉘 기반 easyrule pass 및 webConfigurator 초기화 명령어 등)을 나만의 기술 블로그와 트러블슈팅 노트에 상세히 기록하여 자산화해 두겠습니다.
  • 아키텍처 구조화: OpenVPN 구축 프로세스(사용자 생성 → 인증서 연동 → VPN 서버 세팅 → 클라이언트 배포)를 잊어버리지 않도록 나만의 아키텍처 다이어그램으로 간단히 그려 시각화하겠습니다.

🤝 함께 나누고 싶은 점

“논리적 설정이 모두 맞는데도 통신이 안 된다면, 가장 기초적인 물리적 선 연결(L1)이나 시스템 자체의 숨겨진 디폴트 보안 정책을 의심해 보자!”

에러 메시지(Connection refused, Network is unreachable 등)를 자세히 읽고 Ping을 통해 구간별 단절 위치를 찾아가는 과정이야말로 진짜 실무 역량을 키우는 가장 확실하고 좋은 방법이라는 점을 동기들과 공유하고 싶습니다.
