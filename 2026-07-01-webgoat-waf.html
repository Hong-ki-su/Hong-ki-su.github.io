[웹 모의해킹/WAF] 인프라 보안 수업 - WebGoat 취약점 공격 및 ModSecurity 방어 실무

날짜: 2026년 7월 1일 주제: Kali, webgoat, 패스워드 크래킹, WAF(ModSecurity), 트러블슈팅

📝 오늘 배운 내용 요약

1. 초기 정찰 및 취약점 환경 구축

  • WebGoat 가상 서버 구동: VirtualBox 환경에서 WebGoat 서버(192.168.55.99)를 호스팅하고, 터미널 환경에서 네트워크 인터페이스 정보를 확인하여 점검 대상을 파악했습니다.
  • 프록시 패킷 분석: Burp Suite를 활용하여 브라우저와 웹 서버 간의 통신을 Intercept 하고, JSESSIONID 세션 쿠키 값과 HTTP Request 파라미터가 평문으로 전달되는 구조를 분석했습니다.

2. 시스템 및 웹 애플리케이션 취약점 실습

  • 패스워드 해시 크래킹: 리눅스 /etc/shadow 파일 형태의 md5crypt 해시 패스워드 구조를 이해하고, John the Ripper 도구와 워드리스트(Dictionary)를 활용하여 root 및 계정 비밀번호 해독 실습을 진행했습니다.
  • 웹 폼(Form) 위조 및 악성 서버 구동: 공격자 PC에서 파이썬 내장 모듈(python -m http.server)을 이용해 8000번 포트 웹 서버를 개방하고, 피싱 목적의 HTML(hacking.html) 코드를 작성 및 호스팅했습니다.

3. WAF(Web Application Firewall) 구축 및 방어 실무

  • ModSecurity 연동: Ubuntu 기반 Apache2 웹 서버에 ModSecurity 모듈(security2)을 설치 및 활성화하여 웹 방화벽 환경을 구성했습니다.
  • 방어 룰(Rule) 작성 및 탐지: XSS 공격(<script>alert(document.cookie)</script>)을 시도하고, ModSecurity 설정의 Action 속성을 통해 403 Forbidden으로 접근을 차단했습니다.
  • Audit Log 분석: modsec_audit.log 파일을 실시간으로 트래킹(tail -f)하며 공격자의 IP, Payload, 탐지된 룰(ID: 100002) 정보를 상세히 분석했습니다.

💻 핵심 실습 명령어 및 상세 내용

1. 패스워드 크래킹 (Password Cracking)

리눅스 시스템의 탈취된 해시(Hash) 비밀번호를 해독하는 실습을 진행했습니다. * 사용 도구: John the Ripper (john) * 실행 명령어: sudo john webgoatuser.txt * 실습 내용: md5crypt 방식으로 암호화된 비밀번호 해시 파일(webgoatuser.txt)을 분석하고, /usr/share/john/password.lst 워드리스트(사전 파일)를 활용한 사전 대입 공격(Dictionary Attack) 원리를 확인했습니다.

2. 파이썬 웹 서버 호스팅 및 가짜 페이지 구동

공격자가 악성 스크립트나 가짜 피싱(Phishing) 페이지를 배포하기 위해 간이 웹 서버를 구축하는 과정을 실습했습니다. * 실행 명령어: ```bash # HTML 파일 작성 sudo vi hacking.html

# 파이썬 내장 모듈을 이용한 8000 포트 웹 서버 실행 sudo python -m http.server

실습 내용: 0.0.0.0:8000 포트로 서버를 개방하여 192.168.55.161:8000/hacking.html에 접근 성공했습니다. (GET 요청에 대한 200 OK 로그 확인)

  1. WAF(웹 방화벽) ModSecurity 설치 및 초기 설정 우분투(Ubuntu) 서버 환경에서 아파치(Apache) 웹 서버를 올리고, 대표적인 오픈소스 웹 방화벽인 ModSecurity를 연동했습니다. sudo apt install -y apache2 libapache2-mod-security2 sudo a2enmod security2 초기 설정 복사 명령어:sudo cp /etc/modsecurity/modsecurity.conf-recommended /etc/modsecurity/modsecurity.conf

  2. 아파치 설정 파일 문법 검사 및 서버 재시작 웹 방화벽 룰 적용을 위해 아파치를 재시작하기 전 문법을 검사했습니다.

실행 명령어:sudo apache2ctl configtest sudo systemctl restart apache2

  1. 공격 차단 테스트 및 Audit Log 분석 설정한 WAF가 실제 공격(XSS)을 어떻게 방어하고 기록을 남기는지 실시간 로그로 분석했습니다.

로그 실시간 모니터링 명령어: sudo tail -f /var/log/apache2/modsec_audit.log

방어 성공: 웹 브라우저에서 403 Forbidden (접근 거부) 페이지 반환.

📖 추가 이론: WAF(Web Application Firewall) 및 룰(Rule) 구조 WAF(웹 방화벽)의 개념 일반적인 네트워크 방화벽과 달리 웹 애플리케이션 계층(HTTP/HTTPS)의 트래픽을 심층 분석하여 XSS, SQL Injection 등 악의적인 웹 공격 패턴을 탐지하고 차단하는 보안 시스템입니다.

ModSecurity SecRule 구성 요소 지시어 (SecRule): ModSecurity에게 새로운 보안 규칙을 선언한다는 것을 알리는 시작 키워드입니다.

변수 (Variables): 검사할 대상이 되는 HTTP 트랜잭션의 특정 영역입니다. (예: ARGS 파라미터, REQUEST_COOKIES 쿠키 값 등)

연산자 (Operator): 변수 내의 데이터가 악의적인지 판단하는 매칭 조건이나 정규표현식입니다.

동작 (Actions): 조건이 일치했을 때 방화벽이 취할 행동을 정의합니다. (예: deny 차단, log 로그 기록, status:403 403 코드 반환, msg:‘XSS Attacking’ 로그 메시지 출력)

🛠 주요 트러블슈팅(Troubleshooting) 내역 Q. 설정 파일 복사 및 경로 탐색 에러 (No such file or directory) 원인: /etc/modsecurity/modsecurity.conf-recommended 입력 시 modesecurity로 알파벳 ’e’를 오타 내거나, 경로 탐색 시 ls /var./log/apache2처럼 불필요한 마침표를 입력하여 리눅스가 경로를 찾지 못함.

해결: 리눅스는 대소문자와 철자 하나에도 매우 민감하므로, 명령어 입력 시 Tab 키를 활용한 자동완성 기능을 적극적으로 사용하여 경로와 파일명의 정확도를 높임.

Q. Apache 서버 재시작 실패 (Control process exited with error code) 원인: modsecurity.conf 파일 수정 과정에서 241번째 줄의 변수명 오타(REQUEST_URL -> REQUEST_URI) 및 243번째 줄의 닫는 따옴표 누락(msg:‘XSS Attacking’)과 같은 문법 오류(Syntax Error) 발생.

해결: sudo apache2ctl configtest 명령어를 실행하여 에러가 발생한 정확한 줄 번호(line)와 원인(Missing closing quote 등)을 파악함. 이후 vi 편집기에서 해당 라인(:241, :243)으로 직접 이동하여 오타를 수정하고 서버를 정상적으로 재시작함.

💭 오늘의 회고 📌 배운 점 웹 애플리케이션의 취약점을 공격자의 시선에서 찔러보고, 이를 방어자 입장에서 WAF(웹 방화벽) 룰로 직접 차단해 보는 공수 양면의 과정을 체득했습니다.

특히 방화벽 로그(modsec_audit.log)를 분석하며 내가 던진 Payload가 어떤 근거(Pattern match)로 차단되었는지 가시적으로 확인할 수 있어 보안 솔루션의 작동 원리를 깊이 이해하게 되었습니다.

🚨 어려운 점 및 개선할 점 터미널 실습 중 경로 작성 시 불필요한 마침표를 찍거나(/var./log/), 복사할 원본 파일의 스펠링을 틀리는(modesecurity) 등 사소한 오타로 인해 서비스가 정상 작동하지 않는 문제를 겪었습니다.

ModSecurity 설정 파일 수정 시, 241번 줄의 변수명 오타 및 243번 줄의 닫는 따옴표 누락과 같은 문법 오류(Syntax Error)가 웹 서버 전체의 재시작 실패로 이어지는 치명적인 결과를 경험했습니다.

💡 액션 플랜 설정 파일 수정 작업 전후로는 반드시 sudo apache2ctl configtest와 같은 검증 명령어를 습관화하여 크로스 체크를 진행하겠습니다.

서비스 재시작 실패 에러 발생 시 당황하지 않고, 터미널이 안내하는 systemctl status 명령어나 검증 도구를 통해 에러 로그가 가리키는 정확한 ’Line(줄 번호)’과 원인을 꼼꼼히 읽는 습관을 들이겠습니다.

🤝 함께 나누고 싶은 점 새로운 룰을 적용하거나 설정 파일을 수정할 때는 반드시 apache2ctl configtest와 같은 사전 문법 검증 명령어를 실행하여 크로스 체크를 하는 습관이 인프라 엔지니어에게 필수적임을 깨달았습니다.
